# AI Book Finder

Веб-приложение с chat-like интерфейсом для семантического подбора книг по запросу на естественном языке.

Проект реализует ключевые требования тестового задания:
- загрузка каталога книг через веб-интерфейс;
- индексация книг в open-source векторную БД;
- семантический поиск по смыслу запроса;
- возврат top-5 наиболее релевантных книг;
- разворачивание через `Docker Compose` в одном compose-файле.

---

## Что внутри

- **Frontend**: Streamlit
- **Backend API**: FastAPI
- **Vector DB**: Qdrant
- **Embedding model**: `intfloat/multilingual-e5-small`
- **Контейнеризация**: Docker + Docker Compose

---

## Что умеет система

- загружать каталог книг в формате JSON;
- валидировать структуру входных данных и возвращать понятные ошибки;
- добавлять и обновлять книги в индексе через upsert;
- интерпретировать запросы по жанру, теме, ключевым смысловым сигналам и историческому контексту;
- показывать статус каталога: загружен он или ещё пуст;
- объяснять, почему конкретная книга попала в выдачу.

---

## Почему именно такой стек

### Qdrant
Qdrant — open-source векторная БД, хорошо подходящая для локальных и прикладных сценариев семантического поиска. Она поддерживает быстрый upsert, удобную payload-модель и простой запуск в Docker.

### FastAPI
FastAPI даёт компактный, быстрый и хорошо типизированный backend-слой, что удобно для тестового задания, где важно одновременно показать инженерную аккуратность и быстрый onboarding.

### `intfloat/multilingual-e5-small`
Для задачи поиска по смыслу на русском языке важен баланс между качеством retrieval, скоростью холодного старта и воспроизводимостью запуска в Docker-сценарии. Семейство E5 хорошо подходит для semantic search и retrieval-задач.

### Streamlit
Streamlit позволяет быстро собрать чистый chat-like интерфейс без лишнего фронтенд-шума. Для тестового это даёт хороший баланс: UI есть, им удобно пользоваться, а основное внимание остаётся на AI/search-части.

## Архитектура

Подробное описание архитектуры находится в файле [`docs/architecture.md`](docs/architecture.md).

Коротко:
- пользователь взаимодействует с UI в Streamlit;
- Streamlit вызывает backend API на FastAPI;
- backend принимает каталог, валидирует JSON и вычисляет embeddings;
- embeddings сохраняются в Qdrant;
- при поиске backend кодирует запрос, выполняет semantic search в Qdrant, затем делает лёгкий reranking и возвращает top-5 книг.

---

## Retrieval pipeline

1. Пользователь загружает `books.json` через UI.
2. Backend сохраняет файл и валидирует его структуру.
3. Каждая книга преобразуется в индексируемый документ из названия, автора, жанра, описания и года.
4. Для каждого документа вычисляется embedding с префиксом `passage:`.
5. Векторы и payload книг сохраняются в Qdrant через upsert.
6. При поиске пользовательский запрос нормализуется и интерпретируется: выделяются жанры, темы и ключевые сигналы.
7. Запрос кодируется embedding-моделью с префиксом `query:`.
8. Qdrant возвращает ближайшие по смыслу книги.
9. Backend применяет лёгкий reranking по совпавшим сигналам.
10. Пользователь получает top-5 книг и объяснение, почему они релевантны.

## Запуск

### 1. Требования
- Docker
- Docker Compose

### 2. Запуск проекта

Из корня проекта выполните:

```bash
docker compose up --build
```

После старта будут доступны:
- Frontend: `http://localhost:8501`
- Backend API: `http://localhost:8000`
- Qdrant: `http://localhost:6333`

### 3. Первый запуск

После открытия UI:
1. Нажмите **«Индексировать пример из проекта»**, чтобы загрузить и проиндексировать `data/books.json`.
2. Убедитесь в sidebar, что статус каталога изменился на **«Каталог готов к поиску»**.
3. Введите запрос в чат, например:
   - `историческая проза о революции в России`
   - `мрачная антиутопия`
   - `детектив с расследованием убийства`

### 4. Загрузка собственного каталога

В sidebar можно загрузить свой `JSON`-файл. После загрузки книги будут **добавлены или обновлены** в индексе.

Ожидаемый формат:

```json
{
  "books": [
    {
      "title": "1984",
      "category": "Антиутопия",
      "description": "История о тоталитарном обществе...",
      "author": "Джордж Оруэлл",
      "year": 1949
    }
  ]
}
```

Также поддерживается и список объектов без обёртки `{"books": ...}`.

Если отправить другой формат, backend вернёт понятную ошибку: например, что JSON повреждён, список книг пуст или у записи отсутствуют обязательные поля.

## API

### `GET /healthz`
Лёгкая проверка доступности backend.

### `GET /catalog/status`
Показывает, создана ли коллекция и сколько книг уже проиндексировано.

### `POST /catalog/index-default`
Индексация дефолтного каталога `data/books.json`.

### `POST /catalog/upload`
Загрузка пользовательского JSON-файла и upsert в индекс.

### `POST /search`
Семантический поиск.

Пример запроса:

```json
{
  "query": "историческая проза о революции в России",
  "limit": 5
}
```

## Как устроен поиск

### Индексация
Для каждой книги строится расширенный текстовый документ:
- название;
- автор;
- жанр;
- описание;
- год;
- дополнительные акценты на названии и жанре.

Это повышает шанс, что модель лучше уловит не только описание, но и жанровый и контекстный смысл книги.

### Интерпретация запроса
Перед поиском backend отдельно разбирает пользовательский запрос и извлекает:
- ключевые слова;
- жанры;
- темы;
- тип запроса;
- дополнительные смысловые сигналы.

Эта интерпретация возвращается в ответе и показывается в UI, чтобы сделать логику поиска более прозрачной.

### Embedding pipeline
- для запросов используется префикс `query:`;
- для книжных документов используется префикс `passage:`;
- embeddings нормализуются;
- в Qdrant используется `cosine similarity`;
- поверх dense retrieval применяется лёгкий signal-aware reranking.

### Upsert-логика
Книга получает стабильный идентификатор на основе:
- `title`
- `author`
- `year`

Это позволяет при повторной загрузке каталога не дублировать записи, а обновлять существующие.

### Explainability
Система возвращает не только список книг, но и короткое человеческое объяснение, почему книга попала в выдачу: по теме, жанру, совпавшим сигналам или общему смысловому совпадению.


## Принятые решения и trade-offs

Подробный разбор — в файле [`docs/decisions-and-tradeoffs.md`](docs/decisions-and-tradeoffs.md).

Коротко:
- выбрана multilingual retrieval-модель с хорошим балансом качества и скорости старта;
- выбран Streamlit, потому что он ускоряет delivery и не отвлекает от retrieval-ядра;
- выбран upsert, а не full reindex, потому что это лучше соответствует реальному продуктовому сценарию;
- поверх dense retrieval добавлен лёгкий reranking по смысловым сигналам;
- объяснения результатов сделаны детерминированными, а не LLM-based, чтобы сохранить простоту запуска и предсказуемость.
