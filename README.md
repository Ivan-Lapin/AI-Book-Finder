# AI Book Finder

Веб-приложение с чат-интерфейсом для семантического подбора книг по запросу на естественном языке.

Проект реализует требования тестового задания:
- загрузка каталога книг через веб-интерфейс;
- индексация книг в open-source векторную БД;
- семантический поиск по смыслу запроса;
- возврат top-5 наиболее релевантных книг;
- разворачивание через `Docker Compose` в одном compose-файле.

---

## Что внутри

- **Frontend**: Streamlit
- **Backend API**: FastAPI
- **Vector DB**: Qdrant
- **Embedding model**: `intfloat/multilingual-e5-base`
- **Контейнеризация**: Docker + Docker Compose

---

## Почему именно такой стек

### 1. Qdrant
Qdrant — open-source векторная БД, хорошо подходящая для локальных и production-like сценариев семантического поиска. Она поддерживает быстрый upsert, удобную payload-модель и простой запуск в Docker.

### 2. FastAPI
FastAPI даёт компактный, быстрый и хорошо типизированный backend-слой, что удобно для тестового задания, где важно одновременно показать инженерную аккуратность и быстрый onboarding.

### 3. `intfloat/multilingual-e5-base`
Для задачи поиска по смыслу на русском языке важен баланс между качеством и весом модели. Семейство E5 хорошо подходит для semantic search и retrieval-сценариев.

### 4. Streamlit
Streamlit позволяет быстро собрать чистый chat-like интерфейс без лишнего фронтенд-шума. Для тестового это даёт хороший баланс: UI есть, им удобно пользоваться, а основное внимание остаётся на AI/search-части.

---

## Архитектура

Подробное описание архитектуры находится в файле [`docs/architecture.md`](docs/architecture.md).

Коротко:
- пользователь взаимодействует с UI в Streamlit;
- Streamlit вызывает backend API на FastAPI;
- backend принимает каталог, валидирует JSON и вычисляет embeddings;
- embeddings сохраняются в Qdrant;
- при поиске backend кодирует запрос, выполняет semantic search в Qdrant, затем делает лёгкий hybrid reranking и возвращает top-5 книг.

## Запуск

### 1. Требования
- Docker
- Docker Compose

### 2. Запуск проекта

Из корня проекта выполните:

```bash
docker compose up --build
```

После старта будут доступны:
- Frontend: `http://localhost:8501`
- Backend API: `http://localhost:8000`
- Qdrant: `http://localhost:6333`

### 3. Первый запуск

После открытия UI:
1. Нажмите **«Индексировать пример из проекта»**, чтобы загрузить и проиндексировать `data/books.json`.
2. Введите запрос в чат, например:
   - `роман с неожиданной развязкой`
   - `мрачная антиутопия`
   - `книга про политические интриги в фантастическом мире`

### 4. Загрузка собственного каталога

В sidebar можно загрузить свой `JSON`-файл. После загрузки книги будут **добавлены или обновлены** в индексе.

Ожидаемый формат:

```json
{
  "books": [
    {
      "title": "1984",
      "category": "Антиутопия",
      "description": "История о тоталитарном обществе...",
      "author": "Джордж Оруэлл",
      "year": 1949
    }
  ]
}
```

Также поддерживается и список объектов без обёртки `{"books": ...}`.

## API

### `GET /healthz`
Проверка состояния сервиса и коллекции.

### `POST /catalog/index-default`
Индексация дефолтного каталога `data/books.json`.

### `POST /catalog/upload`
Загрузка пользовательского JSON-файла и upsert в индекс.

### `POST /search`
Семантический поиск.

Пример запроса:

```json
{
  "query": "роман с неожиданной развязкой",
  "limit": 5
}
```

## Как устроен поиск

### Индексация
Для каждой книги строится расширенный текстовый документ:
- название;
- автор;
- жанр;
- описание;
- год;
- дополнительные акценты на названии и жанре.

Это повышает шанс, что модель лучше уловит не только описание, но и жанровый/контекстный смысл книги.

### Embedding pipeline
- для запросов используется префикс `query:`;
- для книжных документов используется префикс `passage:`;
- embeddings нормализуются;
- в Qdrant используется `cosine similarity`;
- поверх dense retrieval применяется лёгкий keyword-signal reranking.

### Upsert-логика
Книга получает стабильный идентификатор на основе:
- `title`
- `author`
- `year`

Это позволяет при повторной загрузке каталога не дублировать записи, а обновлять существующие.

### Chat UX
История чата хранится на стороне Streamlit-сессии. Формально это не полноценный conversational memory-агент, но для тестового задания даёт нужный chat-like UX: пользователь пишет запросы как в чат, а система отвечает списком найденных книг.

## Prompt engineering / AI agent mindset

Хотя задача в первую очередь retrieval-oriented, в проекте специально добавлены элементы, которые показывают мышление AI Agent / Prompt Engineer:

1. **Нормализация пользовательского запроса**
   - очистка и нормализация текста;
   - извлечение ключевых сигналов;
   - попытка интерпретации пользовательского намерения.

2. **Смысловые сигналы**
   - backend выделяет сигналы вроде `мрачность`, `политические интриги`, `магия`, `тайны`, `неожиданная развязка`;
   - затем эти сигналы используются для объяснения результата.

3. **Explainability-first ответ**
   - система возвращает не только список книг, но и краткое объяснение, почему книга была подобрана;
   - это улучшает доверие к результатам и делает ответ ближе к поведению AI-ассистента.

4. **Расширяемость до агентного пайплайна**
   - текущая архитектура уже позволяет добавить query rewriting, reranking, tool-routing или LLM-based explanation без перестройки ядра.

## Принятые решения и trade-offs

Подробный разбор — в файле [`docs/decisions-and-tradeoffs.md`](docs/decisions-and-tradeoffs.md).

Коротко:
- выбрана качественная multilingual retrieval-модель без перегруза инфраструктуры;
- выбран Streamlit, потому что он ускоряет delivery и не отвлекает от retrieval-ядра;
- выбран upsert, а не full reindex, потому что это лучше соответствует реальному продуктовому сценарию;
- поверх dense retrieval добавлен лёгкий hybrid reranking для повышения качества на небольшом каталоге;
- объяснения результатов сделаны детерминированными, а не LLM-based, чтобы сохранить простоту запуска и предсказуемость.

## Примечание по первому запуску

При первом старте backend скачает embedding-модель в локальный cache volume. Это увеличивает время первого запуска, но ускоряет все последующие старты.
