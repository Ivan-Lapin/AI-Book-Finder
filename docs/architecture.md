# Архитектура решения

## Цели архитектуры

При проектировании решения я ориентировался на 4 цели:

1. **Быстрый и понятный onboarding** — запуск одной командой через Docker Compose.
2. **Фокус на retrieval-качестве** — ядро решения должно показывать смысловой поиск, а не просто keyword match.
3. **Инженерная читаемость** — backend, frontend и storage разделены по ролям.
4. **Расширяемость** — архитектура должна позволять безболезненно добавлять reranking, фильтрацию и agent-like поведение.

---

## Компоненты

### 1. Frontend (Streamlit)
Отвечает за:
- чат-интерфейс;
- загрузку `JSON`-каталога;
- отображение статуса backend и каталога;
- показ найденных книг и объяснений.

Почему отдельно от backend:
- UI остаётся простым и независимым;
- backend можно переиспользовать для другого клиента;
- проще масштабировать и тестировать по слоям.

### 2. Backend (FastAPI)
Отвечает за:
- приём и валидацию каталога;
- построение embeddings;
- создание и проверку коллекции в Qdrant;
- upsert записей;
- semantic search;
- интерпретацию запроса;
- подготовку explainable-ответа.

В backend логика разделена на слои:
- `config.py` — конфигурация;
- `models.py` — контракты запросов и ответов;
- `embeddings.py` — работа с embedding-моделью;
- `repository.py` — слой доступа к Qdrant;
- `querying.py` — обработка пользовательского запроса и генерация объяснений;
- `main.py` — API endpoints и orchestration.

### 3. Vector DB (Qdrant)
Отвечает за:
- хранение dense embeddings;
- хранение payload с метаданными книги;
- поиск ближайших соседей по cosine similarity.

Qdrant выбран как practical open-source storage для semantic search, который легко разворачивается локально и поддерживает быстрые upsert-операции.

## Поток данных

### Сценарий 1. Индексация дефолтного каталога
1. Пользователь открывает UI.
2. Нажимает кнопку индексации дефолтного файла.
3. Frontend вызывает `POST /catalog/index-default`.
4. Backend читает `data/books.json`.
5. Каждая книга преобразуется в retrieval-документ.
6. Документ кодируется embedding-моделью.
7. Вектор и payload сохраняются в Qdrant через upsert.

### Сценарий 2. Загрузка пользовательского каталога
1. Пользователь загружает JSON через UI.
2. Frontend отправляет файл в `POST /catalog/upload`.
3. Backend валидирует JSON.
4. Backend приводит элементы к модели `Book`.
5. Backend строит embeddings и делает upsert в Qdrant.

### Сценарий 3. Поиск книг
1. Пользователь вводит запрос в чат.
2. Frontend отправляет `POST /search`.
3. Backend нормализует и интерпретирует запрос.
4. Backend кодирует запрос в embedding.
5. Qdrant возвращает top-k ближайших книг.
6. Backend применяет лёгкий reranking и формирует explanation.
7. Frontend отображает ответ в chat-like виде.

## Схема взаимодействия

```text
Пользователь
   │
   ▼
Streamlit UI
   │ HTTP
   ▼
FastAPI Backend
   ├── Embedding model (Sentence Transformers / E5)
   └── Qdrant
```

## Модель данных

Каждая книга хранится как payload со следующими полями:
- `title`
- `author`
- `category`
- `description`
- `year`
- `document` — агрегированный текст, который реально индексируется

---

## Обработка запросов

В проекте есть лёгкий слой интерпретации запроса:
- нормализация текста;
- удаление части стоп-слов;
- выделение смысловых сигналов;
- классификация запроса по жанру, теме и дополнительным признакам;
- генерация краткого explanation.

Это не заменяет LLM-агента, но показывает инженерный подход к user intent understanding и explainable retrieval.

## Надёжность и эксплуатационные детали

### Состояние сервисов
Для backend используется лёгкий health endpoint, который позволяет быстро проверить доступность API без тяжёлой логики поиска или индексации.

Frontend отдельно запрашивает:
- статус backend через `/healthz`;
- статус каталога через `/catalog/status`.

Это даёт более устойчивое поведение на старте:
- сервисы запускаются по порядку через Docker Compose;
- frontend может показать, что backend ещё запускается;
- backend может честно сообщить, что каталог пока пуст и не готов к поиску.

### Volumes
Используются отдельные volumes для:
- хранения данных Qdrant;
- загрузок пользователя;
- кэша Hugging Face модели.

Это важно, чтобы:
- не терять индекс между перезапусками;
- не скачивать модель заново при каждом старте;
- хранить загруженные пользователем JSON-файлы отдельно от образов.

### Stable IDs и upsert
ID книги детерминированно вычисляется из `title + author + year`. Это обеспечивает повторяемость и предотвращает дублирование при повторной загрузке каталога.

### Предзагрузка embedding-модели
На этапе сборки backend-образа модель `intfloat/multilingual-e5-small` предзагружается в Docker-образ. Это уменьшает риск долгого cold start и таймаутов при первом поисковом запросе.

## Расширяемость

Архитектура изначально заложена так, чтобы безболезненно добавить:
- фильтрацию по метаданным;
- hybrid retrieval;
- cross-encoder reranker;
- LLM-based explanation;
- query rewriting;
- memory и profile-aware подбор;
- API-аутентификацию и rate limiting.

Таким образом, текущее решение остаётся компактным для тестового задания, но при этом не является тупиковым по архитектуре.
